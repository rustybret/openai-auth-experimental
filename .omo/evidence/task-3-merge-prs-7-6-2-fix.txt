NEW (applied fix):
  export function stopOAuthServer() {
    if (oauthServer) {
      const server = oauthServer
      oauthServer = undefined
      server.closeIdleConnections?.()
      server.close(() => {})
      // Force-close any still-active connection (e.g. the in-flight response that
      // triggered this stop) on the next tick, so the response can flush first but
      // the port is released well within the test's 1s wait.
      const forceClose = setTimeout(() => {
        server.closeAllConnections?.()
      }, 0)
      forceClose.unref?.()
    }
  }

DEVIATION FROM PRESCRIBED FIX (important — read this):
The task prescribed a SYNCHRONOUS force-close: call server.closeAllConnections?.()
immediately BEFORE server.close(). I applied that verbatim first and it REGRESSED
6 previously-passing tests with ECONNRESET / ConnectionRefused (30 pass / 6 fail).

Root cause the prescription missed: stopOAuthServer() is invoked from INSIDE the
HTTP request handler in oauth.ts (lines ~341, ~344, ~361, ~364, ~375, ~403) and in
several paths it runs BEFORE/around res.end() flushes. A synchronous
closeAllConnections() destroys the very socket that is mid-response, so the client's
fetch() sees a reset connection. The original code deferred the force-close exactly
to avoid this.

Empirical verification (Bun probe, /tmp/probe2.ts, since removed):
  - orig 25ms deferred force-close: port freed within 1s = true
  - next-tick (setTimeout 0) deferred force-close: port freed within 1s = true
So the deferral interval was never the problem; the broken case was the synchronous
close killing the in-flight response. The minimal correct fix keeps the force-close
DEFERRED (next tick) so the response flushes, while still releasing the port far
inside the 1s expectOAuthPortClosed() window. The reorder (closeIdleConnections +
close scheduled before deferred force-close) plus the test-side 127.0.0.1 bind make
all 4 originally-failing tests pass without regressing the other 32.
